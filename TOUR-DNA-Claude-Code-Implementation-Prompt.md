# TOUR DNA 공모전 제출판 개선 구현 프롬프트

아래 저장소를 기준으로 기존 기능을 보존하면서 공모전 제출판의 데이터 신뢰성, 입력 반영 정확성, 지정과제 적합성, 공개 배포 안정성을 개선하라.

- 저장소: https://github.com/herb39/TOUR-DNA
- 운영 URL: https://tour-dna.lib.lc
- 기본 브랜치: `main`
- 기준 커밋: `9e509ce73633186844969296360c8096086361d5`
- 공모전 지정과제: 7번, 관광 상품 기획 자동화 서비스
- 주 사용자: 여행사/DMC 상품기획자, 지자체·관광재단 실무자
- 핵심 가치: 지역·시기·타깃·운영 조건을 입력하면 한국관광공사 공공데이터를 근거로 지역 관광 DNA를 분석하고, 전략 3안과 편집 가능한 실행안 및 홍보 초안을 생성한다.

## 0. 작업 원칙

1. 먼저 `AGENTS.md`, `CLAUDE.md`, `README.md`, `docs/architecture.md`, `docs/scoring-model.md`, `docs/public-api-status.md`, `docs/test-scenarios.md`, Prisma schema와 기존 테스트를 읽고 현재 구조를 요약하라.
2. 기존 입력 → 분석 → 전략 선택 → 실행안 편집 → 카카오맵 → 인쇄 흐름을 깨뜨리지 마라.
3. 공공데이터에서 확인하지 못한 값이나 API를 추측해 구현하지 마라.
4. 추정 데이터는 데모에 사용할 수 있지만 절대로 `LIVE` 또는 공식 실측값처럼 표시하지 마라.
5. 기능 수보다 데이터 출처와 계산 과정을 심사위원이 이해하고 재현할 수 있는 것이 우선이다.
6. 런타임 LLM은 필수가 아니다. 홍보 문구와 보고서 초안은 우선 결정론적 템플릿으로 구현한다.
7. 한 번에 전부 수정하지 말고 아래 Phase 순서로 구현하며, 각 Phase마다 테스트와 문서를 갱신한다.
8. 기존 사용자 변경을 덮어쓰지 말고, 관련 없는 파일은 수정하지 마라.
9. 사이트 전체 비밀번호 잠금은 제거하고 공개 프로젝트 목록, 작성자 소유권, 프로젝트별 선택적 비밀번호를 적용한다.
10. 런타임에서 Anthropic/OpenAI 등 외부 생성형 AI API를 호출하지 않는다. 점수, 문구, 보고서, 권한 판정은 TypeScript 서버 로직과 PostgreSQL로 처리한다.
11. 무료 플랜 범위를 벗어나거나 결제수단·유료 업그레이드·종량제 사용이 필요한 구현은 시작하지 말고, 이유·예상 비용·무료 대안을 보고한 뒤 중단한다.
12. Claude Code 토큰 절감을 명목으로 보안·테스트·데이터 정확성을 약화하지 않는다. 기계적인 작업은 스크립트·포매터·테스트 러너·CI에 맡기고 Claude에는 설계 판단과 코드 변경만 요청한다.

## 1. 현재 확인된 상태

보존해야 할 구현:

- Next.js 16, React 19, TypeScript, Tailwind CSS
- Prisma 7 + PostgreSQL/Neon
- Vercel 배포 및 `tour-dna.lib.lc` 연결
- 7개 지역 입력
- 관광 DNA 5축 분석
- 전략 3안과 근거 테이블
- 전략 선택 후 실행안 자동 생성
- POI 추가·삭제·순서 변경·날짜 이동
- 시간과 체류시간 편집
- 일정 실행 가능성 경고
- 카카오맵 마커 및 Polyline
- 운영 체크리스트·위험·KPI 편집
- 인쇄/PDF 화면
- Vercel Cron과 공공 API 어댑터
- 단위 테스트 76개 및 E2E 시나리오

확인된 핵심 문제:

- `role`과 `nationality`가 저장·표시만 되고 전략 점수와 산출물에 반영되지 않는다.
- `memo`는 결과에 반영되지 않고, 선호/제외 테마는 문자열 포함 검사 수준이다.
- 추정 `visitorCnt`, 문화자원수요 등의 fixture가 실제 API 성공 스냅샷과 같은 형태로 저장된다.
- `fetchMetricCohort()`가 모든 데이터에 `isSnapshotFallback: false`를 넣어 추정·fixture 축도 `LIVE`가 된다.
- 운영 데모가 `LIVE 5/5`로 표시되지만 Demand와 Network에는 미확인·fixture 값이 섞일 수 있다.
- Network 축은 TourInfo POI 수와 fixture 연관관광지 수를 섞고 출처를 `POI_RELATION` 하나로 표시한다.
- 실 API 동기화 과정에서 `DataSnapshot.rawPayload`가 저장되지 않고 seed에서만 스냅샷을 만든다.
- 동일 데이터 분석에서도 Network의 `collectedAt = new Date()`가 `dataVersion`에 포함되어 키가 달라질 수 있다.
- 반대로 정규화 코호트의 다른 지역 값이 변해도 대상 지역 원값만 같으면 `dataVersion`이 유지될 수 있다.
- DNA 점수는 전국이 아니라 현재 지원 7개 지역 코호트의 상대점수인데 화면 설명이 부족하다.
- 대전 DNA는 유성구 지표, POI는 대전 전체라 행정 범위가 다르다.
- 공개 운영 사이트는 계정·프로젝트 소유권 없이 편집 기능이 노출되고, 사이트 전체 비밀번호는 프로젝트별 공개/보호 요구에 비해 권한 범위가 지나치게 크다.
- 작성자 편집 권한과 일반 열람 권한이 구분되어 있지 않고, 프로젝트별 선택적 비밀번호도 없다.
- 무료 운영 원칙, 무료 한도 초과 시 중단 정책, 런타임 외부 AI 호출 금지 정책이 코드와 문서에 강제되어 있지 않다.
- 현재 npm 11 환경에서 `package.json`과 `package-lock.json` 불일치로 `npm ci`가 실패한다.
- `DATABASE_URL`이 없으면 `next build` 페이지 수집 단계가 실패한다.
- GitHub Actions가 없다.
- 지정과제 7번의 예시 산출물인 다채널 홍보 카피가 없다.
- 입력 조건을 수정하고 안전하게 재분석하는 흐름이 없다.
- 월별 Cron이 고정 `TOUR_DATA_BASE_YM`을 계속 재사용해, 공공 API에 더 최신 기준월이 공개되어도 자동으로 발견·반영하지 못한다.
- DB 저장값을 재사용하는 정책은 있으나 데이터소스별 TTL, 최신 기준월 watermark, stale 상태, 중복 동기화 방지 정책이 없다.

## 2. 서비스 정의, 지정과제 적합성 및 심사 대응

### 2-1. 최종 서비스 정의

이 서비스가 궁극적으로 제공해야 하는 결과를 다음 한 문장으로 고정한다.

> **TOUR DNA는 지역·기간·타깃·운영조건을 입력하면 한국관광공사 공공데이터를 근거로 관광상품 전략, 운영 기획서, 실제 이동을 고려한 코스, KPI와 다채널 홍보 초안을 자동 생성하는 여행사·지자체 실무 도구다.**

관광 DNA 5축 분석은 서비스의 최종 목적이 아니라 다음 산출물을 설득력 있게 만드는 **근거 레이어**다.

1. 입력 조건에 맞는 상품 전략 3안과 추천 이유
2. 선택 전략을 실행할 수 있는 일자별 운영 일정
3. POI, 이동시간, 운영시간, 휴무일을 고려한 일정 위험 경고
4. 역할별 KPI·체크리스트·위험 대응안
5. 채널별로 편집 가능한 홍보 문구 초안
6. 근거 출처·기준월·계산법을 포함한 인쇄/PDF 기획서

사용자 입력과 사용한 데이터 버전이 같으면 결과는 원칙적으로 같아야 한다. 이것은 결함이 아니라 분석 재현성과 심사 신뢰성을 위한 요구사항이다. 입력이 같아도 결과가 바뀌는 경우는 다음으로 제한하고, 변경 사유를 화면에 표시한다.

- 최신 공공데이터 버전으로 사용자가 명시적으로 재분석한 경우
- 역할·국적·선호/제외 테마·기간·예산·이동수단·메모 등 실제 입력이 달라진 경우
- 사용자가 전략이나 POI, 일정, KPI, 홍보 문구를 직접 편집한 경우
- 점수 모델 버전이 변경되고 사용자가 새 모델로 재분석한 경우

### 2-3. 수상 가능성을 높이기 위한 구현 기준

수상을 보장하거나 근거 없는 예상 점수를 문서·화면에 쓰지 않는다. 대신 예비/1차 심사의 구현성·기획성·데이터 적절성·성장성과 최종 심사의 적절성·완성도·실용성·발표력을 다음 완료 기준으로 바꿔 관리한다.

1. **완성도:** 핵심 흐름에서 빈 버튼, 임시 문구, 깨지는 인쇄물, 권한 우회가 없다.
2. **데이터 적절성:** 어떤 수치도 출처·기간·행정범위·가공법 없이 제시하지 않는다.
3. **차별성:** 단순 관광지 추천이 아니라 `분석 → 전략 → 운영안 → 마케팅`을 하나의 실무 산출물로 연결한다.
4. **실용성:** 생성 결과를 편집·저장·공유·인쇄할 수 있고, 실제 경로와 운영시간 위험을 확인할 수 있다.
5. **입력 반응성:** role, nationality, themes, dates, transport, memo가 바뀌면 관련 점수·문구·운영안이 설명 가능한 방식으로 달라진다.
6. **시연 안정성:** 외부 API가 실패해도 검증된 저장본과 명확한 상태 표시로 끝까지 시연된다.
7. **검증:** 가능하면 여행사 실무자 1명과 지자체/관광재단 실무자 1명에게 과업 수행 테스트를 받고, 실제 측정 전에는 시간 절감률·매출 효과를 주장하지 않는다.

대표 시연 데이터는 다음 3개를 seed하되, 각각 입력 차이가 결과 차이로 이어져야 한다.

- 여행사: 제천 1박 2일, 미식·시장 중심 상품
- 지자체: 강릉 가족 체류·소비 확대형 상품
- 외국인 대상: 경주 역사문화·다국어 홍보형 상품

공개 프로젝트 목록은 심사 시 공유와 발견성에는 유용하지만 B2B 실무에서는 기획안 노출 위험이 있다. 현재 MVP에서는 목록에 제목·지역·기간·상태·잠금 여부 같은 최소 메타데이터만 표시하고 본문은 권한에 따라 보호한다. 조직별 비공개나 링크 전용 공개를 추가하는 3단계 공개 범위는 공모전 이후 확장안으로 문서화하고, 이번 범위에는 임의로 넣지 않는다.

### 2-4. 타당성 결론과 중단 기준

- **과제 적합성: 높음.** 7번 과제의 입력 조건, 운영 기획서, 다채널 홍보 카피를 모두 하나의 사용자 흐름으로 직접 구현할 수 있다.
- **기술 구현 가능성: 높음.** 현재 Next.js/PostgreSQL/Vercel 구조와 2인 팀으로 구현 가능하며, 생성형 AI 없이 결정론적 TypeScript 템플릿으로도 핵심 산출물을 만들 수 있다.
- **현재 수상 경쟁력: 보완 필요.** 관광 DNA 분석 화면만 강조하거나 방문자·관계 데이터가 추정값인 상태, 다채널 카피가 미완성인 상태로는 지정과제 완성도와 데이터 신뢰성에서 약하다.
- **보완 후 경쟁력: 충분히 도전 가능.** P0인 provenance, 실제 입력 반영, 운영안, 다채널 카피, 안전한 공유, 3개 시연 시나리오를 완결하고 실무자 검증을 제시하면 단순 관광지 추천 서비스와 뚜렷이 구별된다. 다만 수상 또는 최종 진출을 보장하는 문구는 금지한다.
- **중단 기준:** 핵심 수치를 공식 데이터처럼 보이게 만들기 위해 추정하거나, 유료 API/인프라 결제를 묵시적으로 활성화하거나, 보안 검증을 생략해야만 일정에 맞출 수 있다면 해당 기능 범위를 축소하고 이유를 보고한다.

## 3. 공공 API 부족 사항과 사용자 확인 절차

### 3-1. 구현자가 지켜야 할 원칙

- data.go.kr 신청 상태, Swagger 테스트 결과, 실제 응답 필드와 코드표를 확인하기 전에는 엔드포인트·파라미터·필드 의미를 추측하지 않는다.
- 키가 없거나 활용 신청이 안 된 API는 fixture로 LIVE처럼 흉내 내지 않고 `MISSING`, `SKIPPED` 또는 `ESTIMATED`로 둔다.
- 사용자가 제공하는 샘플에서 `serviceKey`, REST API 키, DB URL, 쿠키와 토큰은 반드시 제거한다.
- 미확인 사항을 발견하면 코드부터 작성하지 말고 `docs/api-verification-request.md`에 필요한 확인 항목, 공식 링크, 최소 샘플 형식과 막힌 기능을 기록한다.
- 공공 API 응답의 이름 문자열을 관계 키로 쓰지 말고 가능한 경우 공식 식별자를 사용한다.

### 3-2. 사용자가 먼저 활용 신청·확인할 API

```text

테스트 날짜:

```

### 3-3. 지표별 해석·개선 규칙

**방문자 수**

- `locgoRegnVisitrDDList`는 통신 기반 일별 방문자 성격의 지표로 다루고, 2박 3일 방문을 서로 다른 3명으로 단정하지 않는다.
- 시도와 시군구 값을 합산하지 않는다. 동일 행정 레벨의 대상 지역과 비교 코호트만 사용한다.
- 기본 분석값은 같은 월의 `평균 일 방문자 수`와 가능하면 `전년 동월 평균 대비 변화율`로 정규화한다.
- 화면에는 단위를 `일평균 방문자/방문일 기준`처럼 명확히 표시하고, 고유 방문객 수로 오해할 표현을 금지한다.
- 샘플과 단위 검증이 끝나기 전 기존 fixture `visitorCnt`는 `ESTIMATED`로 유지한다.

**문화자원 수요 및 기존 Demand 지표**

- `culResDemIxCd`의 공식 코드표를 확보하지 못하면 문화자원 수요를 임의의 코드로 호출하지 않는다. 제공기관 문의가 필요하면 문서에 남기고 해당 지표를 `MISSING` 처리한다.
- 기존 `1101` 값은 일반 관광수요가 아니라 공식 정의가 확인되는 경우에만 `레포츠여행 유형 SNS 언급량`처럼 좁은 의미로 라벨링한다.
- 서로 다른 개념의 지표를 단지 값 범위가 비슷하다는 이유로 하나의 축에 합치지 않는다.

**중심·연관 관광지**

- 2025년 개편 API의 실제 성공 응답을 받은 뒤 연동한다.
- 장소명 문자열이 아니라 `hubTatsCd`, `tAtsCd`, `rlteTatsCd` 등 공식 ID로 POI 관계를 연결한다.
- 연관 관광지 데이터의 공식 제공 기간이 과거로 제한되어 있으면 `CACHED_API` 또는 `기간 한정 공식 데이터`로 표시하고, 2026 실시간 관계라고 표현하지 않는다.
- 공식 API가 없는 기간에는 `CURATED` 관계를 별도로 유지하되 API 관계와 섞어 LIVE로 표시하지 않는다.

**TourAPI 관광정보 상세**

- `KorService2`의 `detailCommon2`, `detailIntro2`, `detailInfo2`, `detailImage2`, `areaBasedSyncList2`를 공식 문서와 실제 샘플로 검증한다.
- 최소 콘텐츠 유형 `12 관광지`, `15 축제/공연/행사`, `32 숙박`, `39 음식점`에 대해 운영시간, 휴무일, 요금, 예약, 주차, 대표 이미지와 이미지 저작권/출처 표시 가능 여부를 확인한다.
- 전국 상세정보를 미리 전부 수집하지 말고, 분석에 선택되거나 후보에 오른 POI만 서버에서 조회해 캐시한다.
- 상세 필드가 없거나 최신성이 불명확하면 `정보 없음`, `운영자 확인 필요`, 수집일을 함께 표시한다.

### 3-4. API 검증 완료 조건

- `docs/public-api-status.md`에 신청 상태, 마지막 성공일, 최신 데이터 기간, 엔드포인트, 단위, 공식 ID, provenance와 미해결 사항이 있다.
- 각 API의 redacted 성공 fixture와 오류/빈 응답 fixture가 테스트 전용으로 저장된다.
- fixture는 운영 데이터베이스에 LIVE 데이터로 seed되지 않는다.
- 사용자 확인이 필요한 API는 `docs/api-verification-request.md`에 남고, 확인 전에는 해당 Phase만 중단하며 독립적인 Phase는 계속 진행할 수 있다.

## Phase 1. 데이터 출처 및 상태 모델 정비 — 최우선

### 1-1. provenance 모델 추가

Prisma에 데이터 출처 상태를 명시적으로 추가하라. 이름은 기존 코드와 충돌하지 않는 범위에서 조정 가능하지만 의미는 다음을 포함해야 한다.

```text
LIVE_API       실제 API에서 이번 동기화로 받은 값
CACHED_API     과거 실제 API 성공 응답을 재사용한 값
CURATED        사람이 검수해 만든 POI/관계 데이터
ESTIMATED      추정·가공 데모 값
MISSING        사용할 수 있는 값 없음
```

- `NormalizedMetric`에 provenance와 가능하면 `snapshotId`를 추가한다.
- `Evidence`에도 분석 당시 provenance를 복사해 보존한다.
- POI와 POI 관계 근거도 출처를 구분할 수 있어야 한다.
- Prisma migration을 작성하고 기존 레코드 마이그레이션 전략을 문서화한다.
- 기존 fixture의 미확인 `visitorCnt`, 문화자원수요는 `ESTIMATED`로 지정한다.
- 실제 API로 확인한 체류·소비·관광서비스수요·다양성은 `LIVE_API` 또는 `CACHED_API`로 지정한다.

### 1-2. 축 상태 계산 수정

- `isSnapshotFallback: false` 하드코딩을 제거한다.
- Evidence provenance를 기준으로 축 상태를 계산한다.
- 축에 `ESTIMATED`가 하나라도 포함되면 `LIVE`로 표시하지 않는다.
- UI 상태는 최소 `실시간 API`, `검증된 저장본`, `큐레이션`, `추정값 포함`, `데이터 부족`으로 사람이 이해할 수 있게 표시한다.
- 전체 상태 `LIVE 5/5` 같은 표현은 실제로 모든 사용 근거가 LIVE일 때만 허용한다.
- 추정값을 점수에서 제외해도 계산 가능한 경우 가중치를 재정규화한다.
- 추정값밖에 없으면 점수를 `null`로 두거나 `참고용 추정`임을 분명히 표시한다.

### 1-3. Network 근거 분리

- 관광지·음식·숙박·체험 POI 수는 `TOUR_INFO` 또는 각 POI의 실제 출처로 표시한다.
- 연관관광지 관계 수는 `POI_RELATION`/`CURATED`로 별도 표시한다.
- 미확인 `POI_RELATION`을 LIVE 출처로 표시하지 않는다.
- Network Evidence를 한 행으로 뭉개지 말고 최소 POI 구성 근거와 관계 근거로 분리한다.

### 1-4. 실제 raw snapshot 저장

- 각 어댑터가 반환한 원본 payload, status, resultCode/resultMsg, itemCount, fetchedAt을 `DataSnapshot`에 저장한다.
- 정규화 지표는 어떤 snapshot에서 만들어졌는지 추적 가능하게 한다.
- API 실패 시 기존 성공값을 삭제하지 말고 `CACHED_API`로 사용할 수 있게 한다.
- fixture가 공식 API 응답처럼 보이는 가짜 `NORMAL SERVICE` envelope를 만들지 않게 구조를 분리한다.

### Phase 1 완료 조건

- 추정값이 포함된 데모에서 `LIVE 5/5`가 나오지 않는다.
- 근거 테이블에 한글 출처명, provenance, 기준월, 행정범위, 수집일, 계산규칙이 보인다.
- API URL 또는 공공데이터 출처 페이지 링크를 제공한다.
- provenance별 단위 테스트를 추가한다.
- 기존 분석 결과는 migration/재분석 후 정상 노출된다.

## Phase 2. 갱신형 DB 캐시와 최신 데이터 자동 반영 — 최우선

DB를 제거하거나 사용자의 페이지 요청마다 공공 API를 호출하지 마라. DB는 분석 재현성과 API 장애 대응을 위한 검증된 캐시로 유지하되, 고정 캐시가 아니라 **최신성 정책이 있는 갱신형 캐시**로 구현한다.

기본 흐름은 다음과 같다.

```text
공공 API
  → 정기 또는 필요 시 동기화
  → DataSnapshot 원본 저장
  → 검증·정규화
  → 새 DatasetVersion 발행
  → 신규 분석은 최신 발행 버전 사용
  → 기존 분석은 당시 버전을 유지하고 재분석 여부를 사용자가 선택
```

### 2-1. 데이터소스별 최신성 정책

모든 데이터를 같은 주기로 갱신하지 말고 데이터 성격별 정책을 둔다. 실제 API 제공 주기를 확인한 뒤 설정값을 조정할 수 있게 하되 기본값은 다음과 같이 구현한다.

| 데이터 | 기본 갱신 정책 |
|---|---|
| 월별 체류·소비·관광서비스수요·다양성 | 매일 최신 기준월 존재 여부만 확인, 새 월 발견 시 전체 동기화 |
| 관광지·음식·숙박·체험 POI 목록 | 7일마다 갱신 |
| 운영시간·휴무일 상세정보 | 7~14일 또는 실행안 검토 시 만료 여부 확인 |
| 축제·행사 | 하루 1회, 개최 임박 데이터 우선 |
| CURATED 데이터 | 자동 덮어쓰기 금지, 운영자 명시적 수정만 허용 |
| 미확인 API | 반복 호출하지 않고 `UNSUPPORTED` 또는 `SKIPPED` 처리 |

다음 메타데이터를 데이터소스·지역·기준월 단위로 저장할 수 있게 스키마를 설계한다.

```text
lastAttemptAt
lastSuccessAt
expiresAt
latestAvailableBaseYm
status
provenance
payloadHash
failureCount
nextRetryAt
```

모델명은 기존 구조에 맞춰 `SourceWatermark`, `DataFreshness`, `SyncCursor` 등으로 정할 수 있다.

### 2-2. 최신 기준월 자동 탐색

`TOUR_DATA_BASE_YM`을 정상 운영의 고정 기준월로 사용하지 마라. 이 값은 자동 탐색 실패 시 운영자가 강제로 지정하는 비상용 override로만 남긴다.

최신 기준월 탐색 함수를 구현한다.

1. Asia/Seoul 현재 날짜 기준 전월을 첫 후보로 만든다.
2. API 호출량이 가장 적고 기준월 존재 여부를 확실히 판별할 수 있는 검증된 지표 API 한 개로 후보 월을 probe한다.
3. 응답이 `SUCCESS`이고 유효 item이 있으면 해당 월을 최신 공개 월로 판정한다.
4. `EMPTY`이면 한 달 전으로 이동한다.
5. 최대 6개월까지만 역방향 탐색한다.
6. 네트워크 오류·인증 오류를 `EMPTY`로 오인하지 않는다. 이 경우 탐색 실패로 기록하고 기존 watermark를 유지한다.
7. 발견된 월이 `latestAvailableBaseYm`보다 새로울 때만 해당 월의 전체 지역·소스 동기화 job을 생성한다.
8. 동일 월을 매일 전체 재수집하지 않는다.

예시:

```text
2026-08 실행 → 202607 probe EMPTY → 202606 SUCCESS
→ 최신 공개 월 202606 유지, 전체 동기화 생략

다음 실행 → 202607 SUCCESS
→ 202607 전체 동기화 시작 → 검증 성공 후 최신 DatasetVersion 발행
```

### 2-3. stale-while-revalidate 조회 정책

서비스 조회는 다음 규칙을 따른다.

- `expiresAt` 이전이면 DB 값을 즉시 사용한다.
- 만료됐지만 과거 성공값이 있으면 과거 값을 즉시 제공하고 상태를 `STALE/CACHED_API`로 표시한 뒤 백그라운드 갱신을 요청한다.
- DB 값이 없으면 동기화 job을 만들고, 짧게 완료를 기다릴 수 있으면 기다린다. 제한 시간 내 완료되지 않으면 데이터 준비 중 또는 데이터 부족 상태를 보여준다.
- API 갱신 실패 시 마지막 성공값을 유지하고 삭제하지 않는다.
- 사용자 페이지 요청에서 6개 API를 동기식으로 모두 호출하지 않는다.
- stale 값을 사용한 분석은 근거와 출력물에 마지막 성공일과 기준월을 표시한다.

Next.js/Vercel 요청 안에서 응답 후 실행이 보장되지 않는 단순 fire-and-forget Promise에 의존하지 마라. Cron, 보호된 sync route, 재시도 가능한 `SyncJob` 중 현재 배포 환경에서 확실히 실행되는 방식을 사용한다.

### 2-4. 중복 호출과 동기화 폭주 방지

`SyncJob` 또는 PostgreSQL advisory lock/고유 제약을 사용한다.

```text
jobKey = sourceCode + regionCode + baseYm + operation
```

- 같은 `jobKey`가 `PENDING` 또는 `RUNNING`이면 새 job을 만들지 않는다.
- 성공·실패·부분성공·재시도 횟수·다음 실행시각을 저장한다.
- 일시적 실패만 지수 백오프로 재시도한다.
- 인증 실패, 잘못된 endpoint, 미지원 API는 자동 재시도하지 않는다.
- 7개 지역×다수 API를 한 서버리스 요청에서 끝까지 순차 실행하지 말고 지역 또는 데이터소스 단위로 배치한다.
- API 일일 호출 한도를 설정값으로 두고 예산을 초과하면 다음 실행으로 넘긴다.

### 2-5. 검증 후 원자적 최신 버전 발행

- API 응답을 받자마자 현재 서비스 데이터에 부분 덮어쓰지 않는다.
- 원본 snapshot 저장 → 스키마 검증 → 정규화 → 필수 지표 검증 단계를 거친다.
- 지역·기준월에 필요한 동기화가 완료되면 하나의 `DatasetVersion`을 발행한다.
- 분석은 `PUBLISHED` 버전만 사용한다.
- 부분 실패 버전은 관리자 화면에서 확인할 수 있지만 기본 분석 버전으로 자동 승격하지 않는다.
- 새 버전 발행 시 이전 성공 버전을 즉시 삭제하지 않는다. 기존 분석 재현과 rollback에 필요한 최소 버전을 retention 정책에 따라 보관한다.

### 2-6. 기존 프로젝트와 최신 데이터의 관계

최신 데이터가 들어와도 기존 `AnalysisResult`와 사용자가 편집한 실행안을 조용히 변경하지 마라.

- 각 분석은 사용한 `datasetVersionId`, `dataVersion`, `baseYm`을 유지한다.
- 현재 발행 버전과 프로젝트 분석 버전이 다르면 분석 화면에 다음과 같은 안내를 표시한다.

```text
새로운 2026년 7월 관광데이터가 공개되었습니다.
현재 분석은 2026년 6월 데이터 기준입니다.
```

- `기존 결과 유지`와 `최신 데이터로 재분석` 선택을 제공한다.
- 재분석은 Phase 6의 안전한 재분석 정책을 따른다.
- 프로젝트 목록에도 `최신`, `업데이트 가능`, `저장 데이터 사용 중` 상태를 표시한다.

### 2-7. 관리자 관측 화면

최소한 다음 정보를 조회할 수 있는 관리자 화면 또는 보호된 운영 리포트를 제공한다.

- 데이터소스별 최신 공개 기준월
- 마지막 시도·성공·실패 시각
- 현재 freshness 상태와 만료시각
- 지역별 수집 건수
- 실행 중·대기 중·실패한 SyncJob
- 다음 재시도 시각과 실패 원인
- 현재 발행 DatasetVersion
- 보호된 수동 probe·동기화·재시도 버튼

### Phase 2 완료 조건

- 고정 `TOUR_DATA_BASE_YM`을 바꾸지 않아도 API에 새 기준월이 공개되면 자동 발견한다.
- 같은 최신 월을 매일 전체 중복 저장하지 않는다.
- API 장애 시 마지막 성공값으로 서비스가 유지되고 UI에 stale 상태가 표시된다.
- 동시에 여러 요청이 들어와도 같은 source/region/baseYm 동기화가 한 번만 실행된다.
- 부분 수집 데이터가 현재 분석 버전으로 노출되지 않는다.
- 새 데이터 발행 후 기존 프로젝트 결과는 유지되고 `최신 데이터로 재분석` 안내가 나타난다.
- probe 성공·EMPTY·네트워크 실패, TTL 만료, stale fallback, job 중복 방지, 버전 발행에 대한 단위·통합 테스트가 추가된다.

## Phase 3. 결정론과 데이터 버전 정확성

1. 동일한 전체 입력 + 동일한 데이터 스냅샷 + 동일한 모델 버전이면 동일 결과와 동일 `analysisKey`가 나와야 한다.
2. `dataVersion`에서 분석 시각 같은 휘발성 값을 제거한다.
3. Network의 수집일은 `new Date()`가 아니라 실제 POI/관계 데이터의 최신 수집·수정 시각을 사용한다.
4. min-max 정규화가 전체 코호트에 의존하므로, 데이터 버전에는 다음 중 하나를 포함한다.
   - 지표별 정렬된 전체 코호트 값 해시, 또는
   - 지표별 코호트 버전/스냅샷 집합 해시.
5. `analysisKey` 입력에 `role`, `nationality`, 실제로 결과에 영향을 주는 모든 입력을 포함한다.
6. 배열 입력은 의미상 순서가 없다면 정렬 후 해시한다.
7. 다음 테스트를 추가한다.
   - 같은 입력/데이터는 같은 결과와 키
   - role 변경 시 산출물 또는 점수/템플릿이 달라짐
   - nationality 변경 시 산출물 또는 체크리스트가 달라짐
   - 코호트 내 다른 지역 값 변경 시 dataVersion 변경
   - 수집과 무관한 현재 시각 변화로 dataVersion이 바뀌지 않음

## Phase 4. 사용자 역할·국적·테마를 실제 결과에 반영

### 4-1. 역할별 산출물

`TRAVEL_AGENCY`:

- 판매 가능한 상품명과 콘셉트
- 핵심 판매 포인트
- 운영 일정 및 예약 체크리스트
- 가격을 자동으로 날조하지 말고 사용자가 원가/판매가를 입력할 수 있는 선택 필드
- 판매 채널과 홍보 문구
- 운영 위험과 대응

`LOCAL_GOV`:

- 정책·사업 목적
- 지역경제 및 체류소비 확대 논리
- 협력 주체와 역할
- 예산 항목 템플릿. 금액은 사용자가 입력하지 않으면 생성하지 않는다.
- 성과지표, 측정 방법, 데이터 출처
- 공공성·지속가능성·지역상생 체크리스트

- 같은 전략이어도 역할에 따라 실행안의 제목, 배경, 체크리스트, KPI, 인쇄 문서 구성이 달라야 한다.
- 역할별 점수 가중치를 바꾸는 경우 `docs/scoring-model.md`에 근거와 공식을 작성한다.

### 4-2. 내국인/외국인 반영

`FOREIGN`일 때 다음을 산출물과 실행안 점검 항목에 반영한다.

- 외국어 안내 가능 여부 확인
- 국제결제·교통·길찾기·예약 방식 확인
- 문화적 설명이 필요한 콘텐츠
- 외국인 방문·국적 다양성 실제 지표가 있을 경우 근거로 연결
- 데이터가 없으면 가능 여부를 단정하지 않고 `운영자 확인 필요`로 표시

`DOMESTIC`은 국내 교통·예약·재방문·지역 소비 관점의 체크리스트를 사용한다.

### 4-3. 테마 입력 개선

- 자유문자열 substring 비교만 사용하지 말고 구조화된 테마 코드를 제공한다.
- 예: 미식, 전통시장, 자연, 웰니스, 역사문화, 축제, 가족, 야간, 청년, 과학, 해양, 체험.
- 템플릿에 지원 테마 코드를 추가해 선호/제외 테마가 명시적으로 점수와 후보 필터에 반영되게 한다.
- 기존 자유 입력은 `추가 요청사항`으로 남기되 자동 반영되지 않은 경우 UI에서 알려준다.

## Phase 5. 지정과제 7번용 다채널 홍보 초안

선택된 전략과 실행안에 `홍보 콘텐츠` 섹션을 추가한다. 우선 결정론적 템플릿으로 구현하며 다음 결과를 만든다.

- 상품 제안서용 3문장 요약
- 랜딩/상세페이지 제목과 본문
- 인스타그램용 짧은 카피와 해시태그
- 블로그용 소개문
- 여행사용 판매 포인트 3개 또는 지자체용 보도자료형 요약
- 각 콘텐츠 복사 버튼

필수 규칙:

- 공공데이터 수치를 쓸 때는 해당 Evidence가 실제로 존재할 때만 사용한다.
- 추정값을 사실처럼 홍보 문구에 넣지 않는다.
- 역할, 국적, 지역, 여행월, 전략명, 선택 POI를 반영한다.
- 홍보 콘텐츠도 인쇄 화면에 선택적으로 포함할 수 있게 한다.
- 생성 결과를 DB에 저장해 사용자가 수정할 수 있게 한다.

## Phase 6. 조건 수정 및 안전한 재분석

- 분석 화면에 `조건 수정` 버튼을 추가한다.
- 기존 입력값이 채워진 수정 폼을 제공한다.
- 저장 후 재분석 전에 기존 선택 전략과 실행안이 오래된 결과가 될 수 있음을 안내한다.
- 재분석은 DB transaction으로 처리한다.
- 새 분석 성공 전 기존 분석을 삭제하지 않는다.
- 성공 시 `selectedStrategyResultId`를 초기화한다.
- 기존 실행안은 사용자의 확인 없이 조용히 덮어쓰지 않는다. `보관 후 새 실행안 생성` 또는 `삭제 후 재생성` 정책을 명시적으로 선택한다.
- 분석 실패 시 사용자가 입력한 수정값과 기존 분석을 안전하게 유지한다.
- 프로젝트 생성 중 분석 실패 시 빈 DRAFT 프로젝트가 중복 생성되지 않게 transaction 또는 재시도 경로를 만든다.

## Phase 7. 비교 코호트와 행정 범위 설명

- 현재 DNA 점수가 `지원 7개 지역 내 상대점수`임을 분석 화면에 표시한다.
- 지표마다 코호트 수 N과 min/max 정규화 범위를 근거 패널에 표시한다.
- 전국 백분위가 아닌데 전국 기준처럼 표현하지 않는다.
- 대전은 `DNA 지표: 유성구 / 추천 장소: 대전광역시 전체`를 눈에 띄게 경고한다.
- 가능하면 대표 데모를 제천·강릉 등 행정범위가 일치하는 지역으로 교체한다.
- 전국 확대는 이 작업의 필수 완료 조건이 아니다. 먼저 표시 정확성을 고친다.

## Phase 8. 사이트 전체 잠금 제거와 공개 프로젝트별 비밀번호 — 최우선 보안 작업

사이트 전체를 하나의 비밀번호로 잠그는 현재 방식은 완전히 제거한다. 홈, 소개, 새 프로젝트 작성 화면과 프로젝트 목록은 공개한다. **모든 프로젝트는 목록에 표시**하며, 프로젝트 생성 시 비밀번호를 설정하지 않았으면 누구나 결과를 열람할 수 있고, 비밀번호를 설정했으면 목록 카드까지만 공개하고 본문은 비밀번호 검증 후 열람하게 한다.

이 정책에서는 URL이 공개 목록을 통해 발견될 수 있으므로 URL 자체는 보안 키가 아니다. URL은 프로젝트 위치 식별자이고, 보호 프로젝트의 실제 열람 수단은 프로젝트 비밀번호다. 보안을 URL의 은닉성에 의존하지 않는다.

### 8-1. 제거할 전역 잠금

- `SITE_ACCESS_PASSWORD`, 전역 로그인 화면, 전역 비밀번호 쿠키, 이를 검사하는 middleware/proxy를 제거한다.
- `.env.example`, 배포 문서, 운영 점검 문서에서 전역 잠금 설정을 삭제한다.
- 홈, 프로젝트 생성, 프로젝트 목록은 로그인 없이 접근할 수 있게 한다.
- 관리자·동기화 route의 `CRON_SECRET` 및 Phase 10의 관리자 인증은 제거하지 않는다.

### 8-2. 공개 범위와 권한 모델

```text
OWNER   작성자. 조회·수정·삭제·재분석·비밀번호 설정/변경/해제 가능
VIEWER  일반 방문자. 비밀번호가 없으면 즉시 열람, 있으면 검증 후 열람·인쇄만 가능
```

- 모든 프로젝트를 공개 목록에 노출한다. 보호 프로젝트도 목록에서 숨기지 않는다.
- 목록 카드에는 프로젝트명, 지역, 여행 시기, 작성 시각, 진행 상태, 비밀번호 보호 여부만 표시한다. 메모, 상세 입력, 전략, 실행안, 복구 코드 등 본문 데이터는 목록 응답에 포함하지 않는다.
- 보호 프로젝트는 잠금 아이콘과 `비밀번호 필요`를 표시하고, 클릭하면 프로젝트 비밀번호 화면으로 이동한다.
- 비밀번호 없는 프로젝트는 `/projects/{publicId}`에서 누구나 바로 읽을 수 있다.
- 프로젝트 작성자는 회원가입 없이 서버가 만든 익명 `OwnerSession`으로 식별한다.
- `OwnerSession` 토큰은 256비트 이상의 암호학적 난수로 만들고 브라우저에는 `HttpOnly`, `Secure`, `SameSite=Lax` 쿠키로 저장한다. DB에는 원문이 아니라 SHA-256 해시만 저장한다.
- 프로젝트 생성 시 현재 `OwnerSession`과 소유권을 연결한다. 같은 브라우저의 작성자는 프로젝트 비밀번호를 다시 입력하지 않아도 편집할 수 있다.
- 쿠키 삭제·브라우저 변경 시 자동 복구할 회원 계정이 없다는 한계를 UI에 설명한다. MVP에서는 프로젝트 생성 직후 1회만 보여주는 고엔트로피 소유자 복구 코드를 제공하고 DB에는 해시만 저장한다.
- 복구 코드는 일반 프로젝트 URL·비밀번호와 분리한다. 복구 성공 시 코드를 회전시키고 새 OwnerSession에 소유권을 연결한다.
- VIEWER는 조회·인쇄만 가능하다. 프로젝트 URL이나 비밀번호를 안다고 수정 권한을 얻지 않는다.
- 고정 공모전 데모도 공개 읽기 전용으로 두고 모든 mutation을 서버에서 거부한다.

### 8-3. 프로젝트 URL과 비밀번호

- `Project.id`는 내부 DB 식별자로 유지하고 외부 URL에는 별도의 충분히 긴 `publicId`를 사용한다. 다만 publicId는 목록에 노출되므로 비밀로 취급하지 않는다.
- 프로젝트 URL은 `/projects/{publicId}` 하나를 사용하고 별도 비밀 shareToken route는 만들지 않는다.
- 사용자가 비밀번호를 설정하지 않은 프로젝트는 인증 세션 없이 공개 본문을 제공한다.
- 비밀번호가 있는 프로젝트는 목록 메타데이터와 비밀번호 입력 화면만 공개한다. 분석 결과, 근거, 전략, 실행안, 인쇄 데이터는 인증 전 서버에서 조회하지 않는다.
- 비밀번호는 Node 런타임의 비동기 `crypto.scrypt` + 프로젝트별 랜덤 salt로 해시하고 `timingSafeEqual`로 비교한다. 일반 SHA-256을 비밀번호 해시에 사용하지 않는다.
- 선택적으로 `PROJECT_PASSWORD_PEPPER`를 서버 환경변수로 사용하되 코드·DB에 저장하지 않는다. pepper 변경 시 기존 비밀번호 재설정 정책을 문서화한다.
- 검증 성공 시 DB-backed VIEWER 세션을 만들고 세션 토큰을 `HttpOnly`, `Secure`, `SameSite=Lax` 쿠키로 저장한다. 세션은 프로젝트별 권한과 비밀번호 버전을 기록하고 기본 7일 후 만료한다.
- OWNER가 비밀번호를 변경하거나 해제하면 기존 VIEWER 세션을 폐기한다. 비밀번호 해제 전 `목록의 모든 사용자가 즉시 열람 가능`해짐을 명확히 확인시킨다.
- 보호 프로젝트 응답은 `Cache-Control: private, no-store`를 사용한다. 비밀번호, 세션 토큰, 복구 코드를 로그·분석 도구·URL query에 넣지 않는다.

### 8-4. 권장 데이터 모델

기존 Prisma 모델명과 충돌하면 이름은 조정할 수 있지만 의미는 유지한다.

```text
OwnerSession
  id, tokenHash(unique), expiresAt, lastSeenAt, createdAt, revokedAt?

Project
  id, publicId(unique), ownerSessionId
  passwordHash?, passwordSalt?, passwordVersion, passwordUpdatedAt?

ProjectAccessSession
  id, projectId, tokenHash(unique), scope(VIEWER)
  passwordVersion, expiresAt, revokedAt?, createdAt, lastSeenAt

ProjectOwnerRecovery
  projectId(unique), recoveryTokenHash, usedAt?, rotatedAt, createdAt

ProjectAccessAttempt
  id, projectId, clientKeyHash, failedCount, windowStartedAt, lockedUntil?, updatedAt
```

- 소유 세션 하나가 여러 프로젝트를 소유할 수 있게 하여 프로젝트마다 쿠키를 추가하지 않는다.
- 프로젝트의 보호 여부는 `passwordHash != null`로 판단하거나 명확한 enum을 사용하되 두 상태가 어긋나지 않게 DB 제약/서비스 함수로 관리한다.
- 사람이 정한 비밀번호는 느린 KDF를 사용하고, OwnerSession·VIEWER 세션·복구 코드처럼 충분한 난수 토큰은 SHA-256 해시만 저장한다.
- 개인정보 최소화를 위해 원본 IP를 저장하지 않는다. 서버 비밀키로 HMAC한 rate-limit용 `clientKeyHash`만 짧은 기간 보관한다.
- 만료된 세션과 오래된 실패 기록을 정기 정리해 무료 DB 용량을 보호한다.

### 8-5. 서버 권한 검증과 남용 방지

- 공개 프로젝트 목록 쿼리와 본문 쿼리를 분리한다. 목록용 select가 실수로 상세 입력·결과·비밀 필드를 반환하지 않게 DTO와 테스트를 둔다.
- 프로젝트 조회, 수정, 삭제, 재분석, 실행안 저장, 인쇄 데이터 조회, 비밀번호 변경 등 모든 route handler/server action에서 중앙 권한 함수를 호출한다.
- 비밀번호 없는 프로젝트와 인증된 VIEWER는 조회·인쇄만 허용한다. 모든 mutation은 OWNER만 허용하며 UI 버튼 숨김에 의존하지 않는다.
- OWNER mutation은 Origin/CSRF 방어와 소유권을 함께 검증한다.
- 비밀번호 실패 제한은 Redis/Upstash 없이 PostgreSQL의 `ProjectAccessAttempt`와 transaction/upsert로 구현한다.
- 예: 프로젝트+clientKey 단위 10분 동안 5회 실패 시 지수 backoff를 적용한다. 한 공격자가 프로젝트 전체를 잠그는 전역 lockout은 만들지 않는다.
- 프로젝트 생성, 비밀번호 검증, 소유자 복구에도 가벼운 DB 기반 rate limit을 적용한다.
- 존재하지 않는 publicId와 비밀번호 오류의 응답은 불필요한 내부 정보를 포함하지 않는다.
- 관리자/cron API는 기존 `CRON_SECRET` 또는 별도 관리자 인증을 유지한다.
- 운영 로그에 API 키, 복구 코드, 비밀번호, 전체 쿠키, 세션 토큰을 출력하지 않는다.

### 8-6. 사용자 흐름과 화면

1. 작성자는 공개 홈에서 프로젝트를 만들고 `프로젝트 비밀번호(선택)`를 설정한다.
2. 서버는 익명 OwnerSession과 프로젝트를 연결하고 작성자를 편집 가능한 결과 화면으로 보낸다.
3. 생성 완료 화면에서 공개 프로젝트 URL, 보호 여부, 1회용 소유자 복구 코드 저장 안내를 제공한다.
4. 모든 프로젝트는 공개 목록에 나타난다. 비밀번호 없는 프로젝트는 클릭 즉시 읽기 화면이 열린다.
5. 보호 프로젝트는 목록에 잠금 상태가 보이며 클릭 시 비밀번호를 입력한다. 성공하면 읽기 전용 결과·인쇄 화면이 열린다.
6. 작성자는 현재 OwnerSession이 소유한 프로젝트에서만 편집 버튼을 보고, 설정에서 비밀번호를 설정·변경·해제한다.
7. 목록 상단에 `프로젝트 이름/지역 검색`, `전체/공개/비밀번호 보호` 필터와 페이지네이션을 제공한다.
8. 새 프로젝트 화면에 `비밀번호를 설정하지 않으면 목록의 누구나 내용을 열람할 수 있습니다`라는 안내와 확인 가능한 비밀번호 표시/숨김 기능을 제공한다.

### Phase 8 완료 조건

- 사이트 전역 로그인/비밀번호 없이 홈, 작성 화면, 프로젝트 목록이 열린다.
- 모든 프로젝트가 목록에 표시되고 공개/비밀번호 보호 상태를 구분할 수 있다.
- 비밀번호 없는 프로젝트는 누구나 열람할 수 있다.
- 보호 프로젝트는 목록에 표시되지만 비밀번호 검증 전 본문·인쇄 데이터를 받을 수 없다.
- 작성자와 일반 열람자의 수정 권한이 서버에서 분리된다.
- 비밀번호·복구 코드·세션 토큰 원문이 DB, 로그, URL query에 남지 않는다.
- 비밀번호 설정/변경/해제, 기존 세션 폐기, rate limit, 소유자 복구가 테스트로 검증된다.

## Phase 9. 무료 운영비 가드와 Claude Code 사용량 최소화

### 9-1. 비용에 대한 사실과 구현 경계

- DB나 서버가 애플리케이션 로직을 실행하도록 해도 그 코드를 작성·검토하는 Claude Code 토큰이 다른 서비스로 이전되지는 않는다. 이 작업으로 줄일 수 있는 것은 런타임 AI API 비용과 불필요한 Claude 반복 작업이다.
- 런타임에는 Anthropic/OpenAI API를 붙이지 않는다. 관광 DNA 점수, 추천 규칙, 홍보 문구, 보고서 초안은 기존 TypeScript 결정론 로직·템플릿·PostgreSQL 저장값으로 처리한다.
- 비용 절감을 이유로 복잡한 PostgreSQL stored procedure/trigger에 핵심 도메인 로직을 숨기지 않는다. DB는 스냅샷, 버전, 세션, rate limit, job lock과 무결성 제약을 담당하고, 변경이 잦은 계산·권한 orchestration은 테스트 가능한 TypeScript 서버 모듈에 둔다.
- 환경변수 `COST_MODE=FREE_ONLY`와 `ALLOW_PAID_SERVICES=false`를 추가한다. 이는 제공자 결제를 기술적으로 차단하는 유일한 장치라고 주장하지 말고, 코드에서 유료 provider adapter와 외부 AI 호출을 거부하는 안전장치로 사용한다.
- 카드 등록, trial 전환, 유료 플랜, 종량제 API, 유료 캐시/큐/모니터링이 필요한 순간에는 구현을 중단하고 사용자 승인을 요청한다. 승인 전에는 패키지 설치·계정 생성·배포 설정 변경도 하지 않는다.

### 9-2. 무료 구성

- 웹/서버: 현재 Vercel Hobby를 유지한다. Hobby 무료 한도 초과 시 서비스가 일시 중지되는 것을 허용하며 Pro로 자동 전환하지 않는다.
- DB: 현재 Neon Free PostgreSQL을 유지한다. 2026-07-22 공식 문서 기준 Free는 프로젝트당 월 100 CU-hours, 0.5GB 저장공간, 월 5GB public network transfer이며 한도 도달 시 compute가 중지된다. 관리자 화면에 DB 크기, 스냅샷 수, 접근 세션 수, 최근 정리일을 표시한다.
- CI/정기 동기화: 저장소가 public인 동안 표준 GitHub-hosted runner의 GitHub Actions를 사용한다. 공식 문서상 public repository의 표준 runner 사용은 무료다. larger runner는 사용하지 않는다.
- 긴 공공데이터 동기화는 Vercel 요청 한 번에 몰지 말고 `scripts/sync-tour-data.ts`처럼 앱 코드와 어댑터를 공유하는 CLI를 만들어 GitHub Actions의 월별 schedule과 `workflow_dispatch`에서 작은 배치로 실행한다. DB와 공공 API 키는 GitHub Actions Secrets에 저장한다.
- Vercel Cron은 Hobby에서 하루 1회, 시간 정밀도 ±59분 제약이 있으므로 짧은 상태 점검/보조 트리거에만 사용한다. 정확한 시각이 필요한 작업을 만들지 않는다.
- public 저장소가 private로 바뀌거나 무료 조건이 달라지면 scheduled workflow를 비활성화하고 수동 동기화로 전환한다. 결제수단이 있는 계정의 초과 사용에 의존하지 않는다.
- Redis/Upstash, 별도 queue worker, 외부 생성형 AI, 유료 지도/경로 API, 이메일/SMS, 유료 오류 추적 서비스를 추가하지 않는다.

### 9-3. 무료 한도 보호

- GitHub Actions에 `timeout-minutes`, `concurrency`, 작은 batch, 재시도 상한을 설정한다. 대용량 artifact를 업로드하지 않고 필요한 경우 retention을 1일로 제한한다.
- API 동기화는 Phase 2의 TTL/watermark를 사용해 새로운 기준월이 없으면 원본 전체를 다시 받지 않는다.
- `DataSnapshot.rawPayload`는 감사·재현에 필요한 필드만 보존하거나 압축/보존기간 정책을 적용한다. 무제한 이벤트·세션·rate-limit 로그를 쌓지 않는다.
- 만료된 `ProjectAccessSession`, `OwnerSession`, `ProjectAccessAttempt`, 실패 job 로그를 주기적으로 정리한다. 프로젝트 원문과 분석 결과는 사용자 삭제 정책 없이 제거하지 않는다.
- 무료 한도 접근 시 관리자 화면에 경고하고 동기화 빈도/보존량을 낮춘다. 한도 초과 시 자동 유료 전환 대신 새 동기화를 중지하고 마지막 검증 저장본을 제공한다.
- `docs/free-cost-policy.md`에 제공자, 현재 플랜, 확인일, 무료 한도, 초과 시 동작, 사용량 확인 위치, 유료 전환 금지를 기록한다. 플랜 숫자는 하드코딩된 비즈니스 로직이 아니라 운영 참고값으로 둔다.

### 9-4. Claude Code 토큰 절감 실행 규칙

- 한 세션에 한 Phase 또는 하나의 좁은 하위 Phase만 구현하고, 통과한 테스트와 남은 일을 짧게 보고한 뒤 중단한다.
- 첫 조사 후 매 Phase마다 저장소 전체를 다시 읽지 않는다. `rg`, 영향 파일, 최근 diff, 관련 테스트만 읽고 필요한 맥락만 유지한다.
- 기존 컴포넌트·유틸·테스트 fixture를 재사용한다. 동일 설명을 README와 여러 문서에 중복 생성하지 말고 정본 문서에 링크한다.
- import 정리, 포맷, Prisma generate, migration diff, lockfile 갱신, 타입검사, lint, 테스트, 빌드는 로컬 명령과 CI에 맡긴다.
- 자동으로 multi-agent/sub-agent를 사용하지 않는다. 병렬 에이전트가 반드시 필요한 경우 예상 이득과 토큰 증가를 먼저 설명하고 사용자 승인을 받는다.
- 막히지 않은 선택사항을 반복 질문하지 말고 문서의 기본값으로 구현한다. 보안 모델, 데이터 삭제, 유료 전환처럼 결과를 크게 바꾸는 질문만 한다.
- 실패 로그 전체를 반복 출력하지 말고 최초 원인과 관련 stack만 요약해 수정한다. 동일 실패 명령을 변경 없이 반복하지 않는다.
- 무료 서비스로 Claude Code 개발 토큰을 대체할 수 있다고 주장하지 않는다. 예상보다 큰 리팩터링이 필요하면 범위를 더 작은 커밋으로 나누고 중단 지점을 보고한다.

### Phase 9 완료 조건

- 런타임 외부 AI API 호출과 관련 키가 없다.
- 유료 provider/queue/cache 의존성이 추가되지 않았다.
- FREE_ONLY 설정에서 핵심 사용자 흐름과 데이터 동기화가 동작한다.
- public 저장소용 CI/월별 동기화 workflow에 timeout·concurrency·수동 실행 경로가 있다.
- 무료 한도 초과 시 결제 대신 중단·stale fallback이 발생하는 정책과 운영 문서가 있다.

## Phase 10. 직접 URL로 접근하는 간단한 관리자 운영 페이지 — P0

운영 오류와 동기화 상태를 DB를 직접 열지 않고 확인할 수 있는 읽기 전용 관리자 페이지를 추가한다.

### 10-1. 접근 경로와 인증

- 관리자 경로는 `/admin/ops`로 고정한다.
- 홈, 헤더, 푸터, 프로젝트 목록 등 일반 사용자 화면에는 관리자 버튼이나 링크를 만들지 않는다.
- sitemap에서 제외하고 `robots`/metadata에 `noindex, nofollow`, 응답에 `Cache-Control: private, no-store`를 적용한다.
- URL을 아는 것만으로 접근할 수 있게 하지 않는다. `/admin/ops` 진입 시 별도 관리자 비밀번호를 요구한다.
- 서버 전용 `ADMIN_ACCESS_PASSWORD`와 `ADMIN_SESSION_SECRET`을 사용하고 둘 다 production에서 필수로 검증한다. query string에 비밀번호를 넣지 않는다.
- 로그인 성공 시 짧은 만료시간(기본 30분)의 HMAC 서명된 `HttpOnly`, `Secure`, `SameSite=Strict` 관리자 세션 쿠키를 발급한다.
- 관리자 로그인 실패에도 PostgreSQL 기반 rate limit을 적용한다. 관리자 비밀번호, 쿠키, secret은 DB나 로그에 저장하지 않는다.
- `CRON_SECRET`, `ADMIN_ACCESS_PASSWORD`, `ADMIN_SESSION_SECRET`은 서로 다른 값이어야 하며 재사용을 감지해 시작 시 경고한다.

### 10-2. 관리자 화면 범위

한 화면에서 다음을 확인할 수 있게 한다.

- 전체 상태 요약: DB 연결, 현재 DatasetVersion, 최신 기준월, 마지막 성공 동기화 시각, 마지막 실패 시각
- 데이터소스별 상태: 마지막 성공/실패, 기준월, itemCount, provenance, stale 여부, 오류 메시지 요약
- 최근 `SyncLog`: trigger, 시작/종료, SUCCESS/PARTIAL/FAILED, 소스별 결과
- 최근 애플리케이션 WARN/ERROR: 오류 코드, 안전하게 정리된 메시지, route 이름, requestId, 발생 시각
- 프로젝트 통계: 전체/비밀번호 없음/비밀번호 보호/상태별 개수. 프로젝트 비밀번호나 상세 사용자 입력은 표시하지 않는다.
- 운영 저장량: DataSnapshot·NormalizedMetric·SyncLog·접근 세션·오류 이벤트 row 수, `pg_database_size(current_database())` 기반 대략적인 DB 사용량
- 만료 세션/실패기록 정리 job의 마지막 실행 시각과 삭제 건수

### 10-3. 오류 이벤트 저장

- Vercel provider 로그 전체를 관리자 페이지에서 가져오려 하지 않는다. 외부 로그 API나 유료 모니터링 서비스를 추가하지 않는다.
- 기존 `SyncLog`는 그대로 사용하고, 사용자 요청 중 운영자가 확인해야 할 WARN/ERROR만 `OperationalEvent`에 제한적으로 저장한다.

```text
OperationalEvent
  id, level(WARN|ERROR), code, safeMessage
  route?, requestId?, contextJson?, createdAt
```

- 공통 `recordOperationalEvent()` helper에서 allowlist 기반 context만 저장한다.
- API 키, 비밀번호, 프로젝트 세션, 복구 코드, 전체 URL/query, 쿠키, DB 연결 문자열, 원본 IP, 민감 사용자 메모를 저장하지 않는다.
- stack trace는 production에서 기본 비저장 또는 짧게 정제하고, 사용자가 입력한 문자열을 그대로 오류 메시지에 합치지 않는다.
- 이벤트는 페이지네이션하고 기본 14일 뒤 정리한다. 같은 오류가 반복되면 fingerprint 기준 집계해 무료 DB를 보호한다.

### 10-4. 읽기 전용 원칙

- MVP 관리자 화면은 조회 전용이다. 프로젝트 수정·삭제, DB 임의 쿼리, secret 노출, 사용자 대리 로그인 기능을 넣지 않는다.
- 동기화 재실행 버튼도 이번 범위에서는 만들지 않는다. 필요 시 기존 보호된 CLI/관리자 API를 별도로 사용한다.
- 모든 관리자 데이터 조회 함수는 서버 전용 모듈에 두고 관리자 세션을 재검증한다. Client Component props나 브라우저 bundle에 secret을 보내지 않는다.
- 오류 목록은 severity, source, 기간으로 필터링하고 서버 페이지네이션을 사용한다.
- DB 연결 실패처럼 페이지 자체 조회가 실패할 때는 secret 없는 최소 오류 화면과 requestId만 표시한다.

### Phase 10 완료 조건

- 일반 사용자 화면 어디에도 관리자 링크가 없고 `/admin/ops` 직접 접근만 가능하다.
- 인증 없이는 관리자 HTML과 데이터 API 모두 401/로그인 화면만 반환한다.
- 관리자 화면에서 최근 동기화 오류와 애플리케이션 WARN/ERROR를 확인할 수 있다.
- 로그/화면/DB에 비밀번호, 세션, API 키, 연결 문자열이 노출되지 않는다.
- `OperationalEvent` 보존기간, 중복 집계, 페이지네이션이 동작한다.
- 인증, rate limit, noindex/no-store, secret redaction 테스트가 통과한다.

## Phase 11. 빌드·CI·동기화 정비

1. Vercel과 동일한 Node/npm 버전을 `.nvmrc` 또는 `engines`/`packageManager`로 고정한다.
2. 해당 버전에서 lockfile을 재생성해 `npm ci`가 깨끗한 환경에서 통과하게 한다.
3. `prisma` CLI는 런타임 필요 여부를 검토해 가능하면 devDependency로 이동한다.
4. GitHub Actions를 추가한다.
   - npm ci
   - prisma generate
   - lint
   - typecheck
   - unit test
   - production build
5. build 단계의 DB 의존을 최소화한다. 불가피하면 CI용 PostgreSQL service와 명확한 환경변수를 사용한다.
6. E2E용 별도 DB를 seed하고 핵심 사용자 흐름을 테스트한다.
7. Phase 2의 최신 기준월 probe, watermark, TTL, stale fallback과 DatasetVersion 발행 흐름을 Phase 9의 GitHub Actions 월별 sync와 보호된 관리자 route에 연결한다. Vercel Cron은 짧은 보조 작업만 담당한다.
8. 7개 지역을 한 서버리스 함수에서 순차 동기화하는 구조의 시간 초과 위험을 줄인다.
   - 지역 또는 데이터소스 단위 배치
   - 재시도 가능한 job 상태
   - 중복 실행 방지
   - 부분 실패 내역 표시
9. 확인되지 않은 방문자수와 POI relation API는 호출 실패를 반복하지 말고 명시적으로 `SKIPPED/UNSUPPORTED` 처리한다.

## Phase 12. 실제 일정·경로 품질 개선 — P1

현재 카카오맵 JavaScript SDK의 `Polyline`은 좌표를 그릴 뿐 경로를 계산하지 않는다. 실제 경로는 별도의 길찾기 REST API에서 거리·예상시간·경로 좌표를 받아 지도에 그려야 한다.

### 12-1. 무료 사용 조건 선검증

코드를 수정하기 전에 2026년 7월 현재 공식 문서와 카카오디벨로퍼스 콘솔에서 다음을 확인하고 `docs/route-api-status.md`에 기록한다.

- 카카오내비 자동차 길찾기: `GET https://apis-navi.kakaomobility.com/v1/directions`
- 자동차 길찾기 무료 일간 쿼터 10,000건, 다중 경유지 길찾기 5,000건
- 카카오맵 대중교통/도보 경로 조회 무료 일간 쿼터 각 1,000건
- 2026년 7월 21일부터 카카오맵 무료 쿼터는 개발자 계정에서 처음 카카오맵 API를 활성화한 앱에만 적용된다는 조건
- 현재 TOUR DNA용 카카오 앱이 `카카오맵 무료 쿼터` 대상인지 여부
- 호출에 필요한 REST API 키, 도메인/IP 설정, 응답 저장·지도 표시 관련 이용약관

**비즈월렛 연결, 유료 API 사용 설정, 카드 등록, 종량제 과금 또는 별도 계약이 필요하면 구현을 시작하지 않는다.** 무료 쿼터 대상 여부와 예상 호출량, 무료 대안을 보고하고 사용자의 승인을 기다린다. 무료 쿼터 안에서는 한도 초과 시 자동 결제되지 않고 429를 폴백 처리하도록 구성한다.

### 12-2. 실제 경로 지원 범위

- 자동차 일정은 카카오모빌리티 자동차 길찾기를 우선 사용한다.
- 하루의 POI가 7개 이하라면 출발지 + 최대 5개 경유지 + 목적지로 한 번에 요청할 수 있다. 그보다 많거나 API 제한에 걸리면 구간별 요청 또는 직선거리 추정으로 안전하게 폴백한다.
- 도보와 대중교통은 카카오맵 REST API의 공식 경로 조회 기능을 실제 성공 샘플로 검증한 뒤 추가한다. 엔드포인트·응답 구조를 추측하지 않는다.
- `MIXED`는 각 구간의 이동수단을 사용자가 선택하지 않은 상태에서 하나의 “실제 경로”라고 단정하지 않는다. 초기 MVP에서는 자동차 구간만 실제 경로를 제공하고 나머지는 추정임을 표시하거나 외부 카카오맵 길찾기 링크를 제공할 수 있다.
- 경로는 “실제 이동 기록”이 아니라 API 조회 시점의 **추천 도로 경로·예상 이동시간**으로 표시한다.

### 12-3. 서버 전용 경로 어댑터와 저장 모델

브라우저에서 카카오 REST API를 직접 호출하거나 REST API 키를 노출하지 않는다. 기존 지도 JavaScript 키와 서버용 REST API 키를 분리한다.

```text
RouteProvider = HAVERSINE | KAKAO_NAVI | KAKAO_MAP
RouteMode = CAR | WALK | PUBLIC_TRANSIT | MIXED
RouteStatus = LIVE_API | CACHED_API | ESTIMATED | UNAVAILABLE

originLat/originLng
destinationLat/destinationLng
waypoints
mode
provider
distanceMeters
durationSeconds
geometry
fetchedAt
expiresAt
requestHash
```

- 서버 전용 `RouteProvider` 인터페이스를 만들고 환경변수/feature flag로 실제 API를 끌 수 있게 한다.
- 자동차 API 요청은 `summary=false`로 받은 `sections[].roads[].vertexes`를 `[lat, lng]` 좌표열로 변환해 카카오맵 Polyline에 그린다.
- 응답의 `summary.distance`, `summary.duration`, 구간별 거리·시간과 조회 시각을 저장한다.
- requestHash는 provider, mode, 정규화한 좌표·경유지 순서, 경로 옵션과 모델 버전으로 만들고 같은 경로의 중복 호출을 막는다.
- 좌표는 캐시 키에 사용할 때 과도한 미세 차이를 제거하되 다른 POI가 충돌하지 않을 정밀도를 테스트로 정한다.
- geometry가 너무 크면 단순화 또는 압축하되 시연 경로가 도로에서 눈에 띄게 벗어나지 않게 검증한다.

### 12-4. 호출·캐시·비용 가드

- 지도 렌더, 페이지 새로고침, 마커 hover, 입력 타이핑마다 경로 API를 호출하지 않는다.
- 사용자가 전략을 선택해 실행안을 처음 생성하거나 `실제 경로 계산/새로고침`을 명시적으로 누를 때만 서버에서 호출한다.
- 기본 경로 캐시 TTL은 30일로 두고, 출발일이 임박했거나 일정/POI 순서가 바뀐 경우에만 새 요청을 허용한다.
- `CACHED_API` 경로는 즉시 보여주고 조회일을 표시한다. 사용자가 새로고침하면 무료 한도 안에서 갱신한다.
- 앱 내부에서 일간 호출 수를 원자적으로 집계하고 공식 한도의 80%에서 관리자 경고, 90%에서 신규 자동 호출 중단, 100% 또는 429에서 Haversine 폴백한다.
- 공식 쿼터가 앱 전체 기준이므로 서버 인스턴스별 메모리 카운터만 믿지 말고 PostgreSQL 집계와 카카오 콘솔 확인 절차를 병행한다.
- 무료 한도를 넘기는 추가 쿼터 설정은 코드·인프라에서 활성화하지 않는다.
- 실패·timeout·429·잘못된 좌표·경로 없음은 일정 생성을 실패시키지 않고 `직선거리 기반 추정`과 외부 길찾기 링크로 폴백한다.

### 12-5. 화면 표시와 실행 가능성 검사

- 실제 경로를 받은 구간은 도로 geometry Polyline, 거리, 예상 이동시간, provider, 조회일을 표시한다.
- 추정 구간은 점선 또는 다른 색으로 구분하고 `직선거리 기반 추정`이라고 지도·일정표·인쇄물 모두에 표시한다.
- 같은 일자 안에 실제 경로와 추정 경로가 섞이면 범례와 구간별 상태를 제공한다.
- 지도에는 `경로 새로고침`을 명시적 버튼으로 제공하고 마지막 조회일과 무료 호출 실패 시 폴백 사유를 표시한다.
- 카카오 REST API 키가 없거나 무료 조건을 충족하지 않아도 기존 직선거리 일정과 지도는 정상 동작해야 한다.
- TourAPI 상세정보 API로 운영시간·휴무일을 확인할 수 있으면 연결하되, 미제공이면 `정보 없음/운영자 확인 필요`로 둔다.
- 실행 가능성 검사는 운영시간, 휴무일, 실제/추정 이동시간 중 확인된 항목과 미확인 항목을 구분해 보여준다.
- `도착 예정 > 운영 종료`, `휴무일`, `이동시간 때문에 다음 일정 겹침`, `경로 미확인`을 서로 다른 경고로 표시한다.

### Phase 12 완료 조건

- 무료 조건을 충족한 경우 자동차 일정에서 직선이 아닌 공식 도로 geometry가 지도에 표시된다.
- 사용자가 같은 일정 페이지를 반복 조회해도 캐시 hit에서는 외부 경로 API 호출이 증가하지 않는다.
- POI 순서 변경 시 requestHash가 바뀌고 새 경로를 요청하거나 명확히 stale 상태가 된다.
- 자동차/도보/대중교통/혼합 모드의 지원·추정·미지원 상태가 거짓 없이 구분된다.
- 키 없음, timeout, 429, 경로 없음에서도 Haversine 추정으로 전체 일정이 계속 표시된다.
- REST API 키가 client bundle, 로그, 응답, DB raw payload에 노출되지 않는다.
- 경로 캐시, 실제 geometry 렌더링, 쿼터 가드, 폴백과 표시 문구 테스트가 통과한다.

## Claude Code 실행 방식 — 전체 1회 검토 후 Phase별 구현

이 문서 전체를 매번 새 대화에 반복해서 붙여 넣거나 한 번에 전부 구현시키지 않는다. 다음의 혼합 방식을 사용한다.

1. 최초 1회에는 이 전체 문서를 마스터 요구사항으로 제공하고 `REVIEW_ONLY` 검토를 시킨다.
2. Claude Code는 코드 수정 없이 현재 코드와 요구사항 차이를 재검증하고 구현 계획·의존성·migration·테스트 범위를 문서로 남긴다.
3. 사용자가 계획을 검토한 뒤 실제 구현은 Phase 하나 또는 원자적 하위 Phase 하나씩 요청한다.
4. 각 구현 대화는 이 마스터 문서, `docs/implementation-plan.md`, `docs/implementation-status.md`와 해당 Phase만 읽는다.
5. 한 Phase가 끝날 때 테스트 결과와 상태 문서를 갱신하고 멈춘다. 다음 Phase를 임의로 시작하지 않는다.
6. 기계적 검색·포맷·테스트는 도구에 맡기고, 관련 없는 파일 전체를 읽거나 장문의 중간 설명을 생성해 토큰을 낭비하지 않는다.

7. 사용자의 API 신청·샘플·코드표 확인이 필요한 내용은 docs/api-verification-request.md에만 모아라. 값이나 URL을 추측하지 마라.

## 첫 번째 실행 지시

지금은 **최초 검토용 프롬프트만 실행**하고 바로 모든 코드를 수정하지 마라.

1. 현재 저장소 상태와 기준 커밋을 확인한다.
2. 위 문제들이 현재 코드에도 존재하는지 파일과 함수 단위로 재검증한다.
3. Phase 8과 Phase 10을 각각 원자적으로 배포 가능한 작은 커밋으로 먼저 분해한다. 전역 잠금만 제거하고 프로젝트 권한 검증이 빠진 중간 상태를 production에 배포하지 않는다.
4. Phase 1~4도 작은 커밋 단위로 분해하며, 보안 작업 이후 Phase 1과 Phase 2를 다른 데이터 기능보다 먼저 완료한다.
5. 프로젝트 소유/비밀번호/접근 세션/운영 이벤트에 필요한 Prisma migration과 기존 프로젝트의 초기 공개 정책을 먼저 제안한다. 기존 프로젝트는 비밀번호 없는 공개 프로젝트로 전환하되 사용자 확인 없이 임의 비밀번호를 만들지 않는다.
6. 공모전 기능 매핑, API 확인 요청과 경로 API 무료 조건 확인을 각각 지정 문서에 남긴다.
7. 위험하거나 선택이 필요한 부분만 질문한다.
8. 검토 문서 작성 후 멈추고 사용자 승인을 기다린다.
9. 계획 승인 후 권장 구현 순서는 Phase 8 → Phase 10 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6 → Phase 7 → Phase 11 → Phase 12이다. Phase 9의 비용 가드는 모든 Phase에 상시 적용한다.
